快捷组合Everything+AutoHotkey使用小记

继上次看的善用佳软在中科大的报告，除了主要的TC介绍外，还额外介绍了几个提高效率的软件，其中就有Everything和AutoHotkey。

Everything设置相对简单，请参看善用佳软的详细介绍“Everything:速度最快的文件名搜索工具”。除了首次打开时，略显卡顿之外（项目工程文件超多，复制任何一个工程都在小时级别，见谅！），之后再打开都是所见所得，速度嗖嗖的。当然安装中文包，看的会顺眼点。

重点说下AutoHotkey，先看善用佳软的介绍“AutoHotkey 0级入门教程:让重复工作一键完成”。安装完成后运行AutoHotkey.exe（还有AutoHotkeyA32.exe和AutoHotkeyU32.exe），右键选择Edit This Script，打开一个名叫AutoHotkey.ahk的文件（是否想到此时可以用Everything搜索到这个文件，此文件在我的文档目录下）。简单翻译下里面的注释：

这个script（指AutoHotkey.ahk）的文件名和路径都是特殊的，因为当你直接启动程序时，会自动加载这个文件。同样任何以.ahk结尾的文本文件都可以通过双击来加载。你可以在任意目录下创建任意多的.ahk文件。你也可以同时运行多个.ahk文件，每个文件都会有各自的托盘图标。可以理解为，将自己定义的快捷键方式放在该文件的后面，因为每次启动AHK都会加载该文件，而不选择重新新建一个，每次都双击去打开运行，当然测试阶段可能那样更方便。好，说重点：组合键快速打开Everything，比如Win+a,要求当Everything已处于激活状态（指在最上层），最小化Everything窗口；当Everything最小化或者在后台或者根本没启动时，打开并激活。根据这个要求，找了N多博文，甚至Google，仍没得到有效解决，最后只能硬着头皮看AHK英文帮助文档，发现不过几分钟就理解并实现了（不是炫耀，英文帮助文档内容全面，且思路清晰，只要耐心点很快就能找到想要的）。

大改动（单栏->双栏）

用了一个晚上加一个早上的时间，按照typecho主页的双栏样式，做了大改动，把原先的单栏硬是变成了双栏。现在都快睁不开眼了，好累。改完却发现，还是原来的单栏来的好看，犯贱！

主题背景图片一直没有满意的，几天前觉得非常适合的图片，过几天后却觉得碍眼了。还有继续阅读的图标，换了一个又一个，看哪个都不顺眼。昨天看了善用佳软xbeta 2012年在中科大作的关于TC的报告，有一点挺有同感的，他说：“简约而不简单的境界是很难达到的。我对网站的要求就是简陋，甚至是丑陋，这个很容易达到。简约的话，总觉得哪里不太够，要调一下，追求美观。而承认简陋的话，你就安心了。”我的理解是，界面有三境界：美观合理；简约不简单；通俗易懂。美观合理是指，既具有视觉上的美观或者冲击力，又不乏合理设计之处，而不是一味的炫，甚至产生反感。简约不简单指的是，视觉上看起来很平淡，说不上美也不觉得丑，却处处流露出设计之用心。通俗易懂，就是没有什么视觉上的效果，但把要表达的内容都用很浅显的方式表达出来了。而我现在就是不断看不顺眼的处境，自己既没有设计师的水平，却硬要挑剔图标，背景图片，动画等为视觉上加分的地方，到头来就是不断看不顺眼。所以干脆放弃一切主题的折腾，继续阅读就用文字输出，背景图片换为纯色背景，反倒觉得有几分美观了。

之所以要改动单栏，是突然觉得没有侧边栏会对阅读造成不方便。按照简洁的标准，看中了typecho主页的侧边栏，于是花了大力气折腾到自己的主题里，原先的单栏变为了双栏。回头一想，侧边栏放什么呢，再看看别的博客，发现侧边栏基本都沦为了广告繁殖地。而且比起单栏，显得内容繁杂。大力气的折腾，到头来还是没觉得好看，好看这个词太主观了。

这么折腾，太耗心力了，关键还没收获。今后不再一时冲动，大改博客主题了，主题就这么滴了，要改也是小添小补。所谓传承，就是可以让人一看某个细节就知道这是某某的东西，博客也是这样，有自己的风格，并且一直传承下去。

Replacing VGA, GOP

implementation for UEFI

各个厂家生产的显示器千差万别，PC行业为了解决这个问题，会在每个显示器上都会附带一块I2C的存储芯片，这个芯片的目的就是来记录我们的显示器的厂商，型号，支持的分辨率等等信息。所以我们从上面的GOPdriver的原型可以看到里面并没有涉及到这些信息，那么这些信息是怎么获取的的呢？我们有另外一个protocol来解决这个问题，那就是EDID Discovered Protocol/EDID Override Protocol；当然现在有些笔记本，一体机之类的由于显示屏幕是预先知道的，那么我们就可以把EDID的相关信息记录在我们的system BIOS里面，这样虽然是减小了系统的兼容性能，但是能节省一点成本。 <!--more-->

无论Panel中有EDID还是通过VBT配置的，都可以通过这个protocol获取EDID信息。

我们知道VBIOS 是计算机图形卡或集成图形控制器的 BIOS，类似系统 BIOS 提供了一组软件所使用的程序函数来访问系统硬件一样，VBIOS 也是提供了一组视频相关的程序使用的函数访问视频硬件。通常由显卡厂商提供一个二进制 Bin 文件给 ODM 厂商，将其包在系统 BIOS 中。

GOP (Graphic Output Protocol) driver 是 EFI 架构下取代传统 VBIOS黑箱的 EFI driver，它被设计成在操作系统启动之前支持基本需求的图形输出功能。也就是说在未进入OS之前就可以进行驱动了。

VBT（Video bios table）是一个特别制作的客制化过得特殊二进制数据块。靠 BMP(BIOS Modification Program) utility 来编辑。VBT 同样如同 legacy vbios 一样记录了 detail timing、GPIO Pins、Clock 等显示方面的参数信息。

Beyond BIOS, Chapter 12, Difference between DXE Driver and EFI Driver

有两类DXE driver：第一类，DXE阶段的很早期执行的DXE driver。这些DXE driver按照依赖关系顺序执行。它们通常包含一些基础 services，CPU、chipset、platform 的初始化代码。同时还产生DXE Core所需要的protocol，用于完成EFI Boot Services和EFI Runtime Service。

第二类DXE driver是那些按照EFI Driver Model。它们不对硬件进行操作，只是在handle base中注册一个Driver Binding Protocol，用于在BDS阶段将driver连接到实际的设备上。

GOP Driver solution

GOP driver按照UEFI Driver model，产生以下protocol：

EFI_DRIVER_BINDING_PROTOCOL，挂在GOP driver本身的image handle上。在调用Start()过程中，GOP driver初始化显示设备，生成child handles并为child handles install所需的protocol：EFI_DEVICE_PATH_PROTOCOL、EFI_GRAPHICS_OUTPUT_PROTOCOL、EFI_EDID_DISCOVERED_PROTOCOL、EFI_EDID_ACTIVE_PROTOCOL、GOP_DISPLAY_BRIGHTNESS_PROTOCOL

PLATFORM_GOP_POLICY_PROTOCOL，GOP driver用于获取VBT和平台信息（比如lid状态）。

GOP_DISPLAY_BRIGHTNESS_PROTOCOL，GOP driver把它install在可以亮度控制的child handles上。

GOP Driver使用VBT来配置。GOP driver在第一次调用Start()时候，通过GOP policy protocol提供的GetVbtData()获取VBT。
